# Configuration

## heramb.config.json

Generated by `heramb init`. See [`heramb.config.example.json`](../heramb.config.example.json) in the repo root.

The config describes:

- Which providers host frontend, backend, and Redis
- Service names and build settings
- Env var mappings between services

## Credentials

Provider tokens are stored outside your project:

```
~/.heramb/keys.json
```

Never commit this file. The CLI reads it during `heramb deploy`. The web dashboard stores tokens in the browser session after sign-in.

## API environment

Copy `packages/api/.env.example` to `packages/api/.env`:

| Variable | Required | Description |
|----------|----------|-------------|
| `JWT_SECRET` | Yes (prod) | Secret for signing JWTs |
| `WEB_URL` | No | Frontend URL (default: `http://localhost:5173`) |
| `API_URL` | No | API URL (default: `http://localhost:3001`) |
| `GITHUB_CLIENT_ID` | No | GitHub OAuth (optional) |
| `GITHUB_CLIENT_SECRET` | No | GitHub OAuth (optional) |
| `GOOGLE_CLIENT_ID` | No | Google OAuth (optional) |
| `GOOGLE_CLIENT_SECRET` | No | Google OAuth (optional) |
| `OPENAI_API_KEY` | No | Optional deploy error diagnosis |

## Dependency graph

Heramb resolves env vars in topological order:

```
REDIS_URL           ← Redis provider (first)
BACKEND_URL         ← backend deploy (needs REDIS_URL)
WS_URL              ← derived from BACKEND_URL (https → wss)
CORS_ORIGIN         ← frontend deploy URL (patched after frontend)
NEXT_PUBLIC_API_URL ← alias of BACKEND_URL
NEXT_PUBLIC_WS_URL  ← alias of WS_URL
```

## Provider locks

Each platform implements a standard interface:

- `deploy()` → `{ url, serviceId }`
- `getStatus()` → `{ status, url }`
- `setEnvVars()` → `{ ok }`
- `getLogs()` → stream
- `destroy()` → `{ ok }`

**Implemented:** Railway, Vercel, Upstash  
**Planned:** Render, Netlify, Fly.io, Heroku, Cloudflare Pages
